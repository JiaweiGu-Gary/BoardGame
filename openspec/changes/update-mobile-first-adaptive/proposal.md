# 提案：Mobile-First 自适应规范与 Cardia 迁移

## Why（为什么要做）

当前项目以桌面端体验为主，移动端在以下方面容易出现问题：

- 核心游戏区比例与可视高度不稳定（浏览器地址栏、刘海安全区、横竖屏切换导致抖动/溢出）
- 触控热区偏小（卡牌/按钮命中困难，误触率高）
- 横竖屏布局缺少统一切换策略（同一组件在不同设备下表现不一致）
- 资源在移动端加载成本过高（首屏大图/音频/动效一次性加载导致白屏或卡顿）

目标是把项目升级为 **“手机端体验为主导，PC 体验为辅”** 的移动端优先架构，并能在未来扩展到 **100 个桌游**，让 AI Agent 可以依据明确规范对每个游戏完成移动端移植。

## What Changes（要改什么）

本变更将引入一个新的能力规格（capability）：`mobile-adaptive`，覆盖全项目的移动端适配规范与落地基建：

- 核心游戏区比例/安全区/滚动策略的统一约束
- 触控热区与可点击性判定的统一规范
- 横竖屏布局切换的 UI 架构约定与断点策略
- 资源分级加载与“移动端预算”约束（图片/字体/音频/动效）
- E2E 测试策略：用标准 viewport 组验证“无整页缩放/无明显溢出/关键热区可点击”
- “移植流程” Skill：让 AI Agent 按固定步骤迁移每个游戏

落地顺序：

1. 先建立规范与 Skill（项目级）
2. 按规范迁移 `Cardia`（作为参考实现 + 测试基线）
3. 以 Cardia 为样板，迁移其它游戏

## Non-Goals（不做什么）

- 不在本提案内统一重做设计风格（仅建立适配约束与基础交互规范）
- 不强制重写所有历史页面（优先覆盖游戏对局页、游戏列表与进入链路）
- 不引入外部大型 UI 框架作为适配依赖（避免增大维护成本）

## Impact（影响评估）

- 可能需要调整部分对局页布局/容器结构，避免依赖 `transform: scale()` 的整页缩放。
- 可能需要为部分 UI 组件增加 `data-testid` 与更一致的可点击热区。
- 资源加载策略变化可能影响首屏与缓存行为，需要配合 E2E 进行回归。

## Rollout（落地与验收）

验收以移动端为主：

- iPhone 主流尺寸（390x844、430x932）竖屏：核心区不被遮挡、无整页纵向滚动
- iPhone 横屏（844x390、896x414）：战场与手牌均可用，允许少量浏览器 UI 偏差
- iPad 竖屏（768x1024）与横屏（1024x768）：布局稳定
- PC：布局不退化（可接受次优，但功能完整）

验收方式：

- Cardia 作为 reference：新增/增强 E2E 覆盖关键 viewport 组
- 输出“移动端适配清单（Skill）”用于后续 100 游戏迁移
